Overview

Cacheer Monitor is a local dashboard that watches a JSONL events log emitted by Cacheer PHP clients. It visualises cache traffic, latency, drivers, and key usage in near-real time so you can debug workloads or showcase Cacheer behaviour.

Core Features

  • Live metrics (hits, misses, puts, flushes, latency percentiles).
  • Event stream explorer with filtering by namespace, type, and key.
  • Drivers distribution donut, namespace breakdown, top key leaderboard.
  • Lightweight JSONL ingestion without external dependencies.

How It Works

  • Cacheer adapters append events to a JSONL file on disk.
  • The monitor exposes a small HTTP API backed by that file.
  • The SPA polls (and optionally streams) metrics + events.
  • CLI helpers manage the dev server and example scenarios.

Quick Start

Spin up the local dashboard in under a minute.

  1. Install dependencies: composer install.
  2. Launch the server: 
    $ php bin/cacheer-monitor serve --host=127.0.0.1 --port=9966
  3. Optionally point to a custom events file before starting: export CACHEER_MONITOR_EVENTS=/path/to/cacheer.jsonl.
  4. Visit http://127.0.0.1:9966 in your browser.

Populate Sample Data

Use the bundled scenarios to emit events covering every driver operation:

$ php Tests/scenarios.php
$ php Tests/scenarios_advanced.php  # load heavier traffic + error cases

Each run appends to the configured JSONL file; use the dashboard “Clear” button or the API to rotate the file.

API Reference

REST endpoints power the SPA and provide a friendly surface for tooling or scripts.

REST Endpoints

GET /api/health Cache health

Simple liveness probe. Returns { "ok": true } when the server is reachable.

GET /api/config Runtime config

Reports the events file path, how it was resolved (env, .env, default), and other runtime hints.

{
  "events_file": "/tmp/cacheer-monitor.jsonl",
  "origin": "env"
}
GET /api/metrics Aggregated stats

Summarises all events into counters, hit-rate, and latency percentiles.

Query
namespace (optional)
Default limit
500 recent entries
{
  "hits": 1200,
  "misses": 45,
  "hit_rate": 0.963,
  "latency": { "avg_ms": 3.1, "p95_ms": 7.8, "p99_ms": 15.2 },
  "drivers": { "redis": 1090, "array": 155 },
  "namespaces": { "default": 800, "critical": 290 },
  "top_keys": { "users:42": 120, "orders:recent": 56 }
}
GET /api/events Event log

Returns the newest events first. Filter by namespace, type, or match portions of the cache key.

Query
limit (default 200), namespace (optional)
{
  "type": "put",
  "ts": 1714310556,
  "payload": {
    "key": "users:42",
    "namespace": "default",
    "driver": "redis",
    "ttl": 300,
    "duration_ms": 5.4
  }
}
POST /api/events/clear Destructive

Rotates the events file: current file is archived with a timestamp suffix and a fresh empty file is created. The dashboard “Clear” button uses this endpoint.

Use only in local/dev environments. Returns { "ok": true, "rotated_to": "..." }.

Server-Sent Events

GET /api/events/stream Realtime ping channel

Provides a lightweight heartbeat used by the SPA to refresh the event list as new lines land in the JSONL file. Messages are simple data: ping payloads.

When EventSource is not available, the dashboard falls back to interval polling using the configured refresh rate.

Configuration & Environment

The monitor favours convention, yet a few knobs keep it flexible during development.

Events file resolution

  1. OS environment: CACHEER_MONITOR_EVENTS.
  2. .env file: read from project root (see src/Support/Env.php).
  3. Fallback: sys_get_temp_dir()/cacheer-monitor.jsonl.

Relative paths are resolved against the repository root.

Refreshing data

  • The UI refresh interval defaults to 5 s (adjust via the select in the header).
  • Manual refresh is available using the dashboard “Refresh” button.
  • Events stream uses SSE ping fallbacks when supported.
  • Large files: consider pruning via POST /api/events/clear after archiving.

Event payload format

Each line in the JSONL file is a single Cacheer event:

{
  "type": "hit",
  "ts": 1714310556,
  "payload": {
    "key": "users:42",
    "namespace": "default",
    "driver": "redis",
    "size_bytes": 2048,
    "duration_ms": 1.9,
    "tags": ["groupA", "beta-testers"]
  }
}

Additional fields (e.g., ttl, exception) appear depending on the event type.

Common Workflows

Practical recipes for daily debugging or demos.

Reset the playground

  1. Run POST /api/events/clear (or click “Clear”).
  2. Replay Tests/scenarios.php.
  3. Open the dashboard; set refresh to 1s for live demos.

Investigate a slow key

  • Filter events by key fragment using the “filter key” input.
  • Inspect latency badges in the event timeline.
  • Cross-check namespaces and drivers to spot anomalies.

Compare namespaces

Use the namespace filter to isolate traffic and export screenshots for reports. Pair with the donut + cards to quantify volume.

Automate reporting

Hit the metrics endpoint from CI to capture regression budgets, or pipe JSONL snapshots into analytics pipelines for longer-term comparisons.

Troubleshooting

When the dashboard looks empty, check these first:

  • Events file path: verify via GET /api/config and ensure the file exists.
  • File permissions: the PHP process must read & write the JSONL file.
  • Large logs: rotate with the clear endpoint if the UI becomes sluggish.
  • PHP version: requires 8.1+ with JSON extension enabled.
  • Browser caching: hard-refresh after updating assets.
